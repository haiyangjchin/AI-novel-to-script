# AI 小说转剧本工具 — 视频讲解稿

> 时长约 8-10 分钟 | 语速中等 | 适合技术展示/项目分享

---

## 开场（0:00 - 0:40）

**【画面：项目首页 / 标题页】**

大家好，今天我来展示一个自己做的项目——**AI 小说转剧本工具**。

简单来说，它的作用就是：**把小说章节自动转换成结构化的剧本格式**。你贴一段小说进去，它调用大语言模型，输出 YAML 格式的剧本，包含完整的角色表、场景划分、对白和动作描写。

对于小说作者、编剧或者影视从业者来说，这个工具可以帮助快速产出剧本初稿，省去大量手工整理的时间。

---

## 项目背景与痛点（0:40 - 1:30）

**【画面：小说 vs 剧本对比图】**

为什么做这个工具？

写小说和写剧本是两种不同的思维方式。小说有大量的心理描写、叙述性旁白，而剧本需要的是——**可视化的动作、可听见的对白、明确的场景转换**。

一个作者如果想把小说改编成剧本，传统做法是：
1. 通读全文，手动提取每个场景
2. 整理所有出场角色和人物关系
3. 把叙述性文字转换成对白和动作指令

这是一个极度耗时的工作。而大语言模型恰恰擅长这种"格式转换 + 结构化提取"的任务。所以这个项目的核心思路就是——**用 AI 来自动化这个流程，把作者从繁琐的格式工作中解放出来**。

## 技术栈选型（1:30 - 3:00）

**【画面：技术栈脑图或列表】**

接下来是技术栈选型，我来说说为什么选这些技术。

### 后端：Python + FastAPI

**FastAPI** 是我后端的核心框架。选它的原因有几个：
- **异步原生**：Python 异步支持最好的 Web 框架之一，LLM 调用是 IO 密集型的，异步能最大化吞吐
- **SSE 流式支持**：LLM 生成内容需要流式输出，FastAPI 的 `StreamingResponse` 原生支持 `text/event-stream`，实现起来非常干净
- **Pydantic 集成**：请求体和响应体的类型校验开箱即用，前端传什么、后端返回什么都清清楚楚
- **性能**：在 Python Web 框架里属于第一梯队

### LLM 接入：OpenAI 兼容 SDK

底层用 `openai` Python 包，但连接的 **base_url** 配置成了 DeepSeek 的 API 端点，也可以换成任何兼容 OpenAI 接口的模型。这样做的好处是——**模型无关**。你可以在 DeepSeek、通义千问、GPT 之间自由切换，只要改一行环境变量。

### 数据格式：YAML

YAML 是整个项目的核心设计决策。为什么选 YAML 而不是 JSON？
- **可读性**：YAML 不需要引号和括号，作者直接用文本编辑器就能修改
- **注释支持**：YAML 原生支持 `#` 注释，方便编剧添加备注
- **Git 友好**：按行缩进，diff 非常清晰
- **行业习惯**：剧本工具常用缩进式结构，YAML 的语法天然契合

### 前端：单文件 HTML

前端没有用任何框架——React、Vue 都没用。就是一个纯粹的**单文件 HTML**，CSS 和 JS 全部内联。为什么这么做？
- **零构建步骤**：不需要 npm install，不需要 webpack，后端起起来就能访问
- **部署简单**：就一个文件，FastAPI 的 StaticFiles 一挂载完事
- **SSE 原生消费**：浏览器的 EventSource 或者 fetch ReadableStream 可以直接消费后端的流式响应
- **够用**：这个工具的核心交互是"输入→转换→展示"，用纯 JS 完全够用，犯不上上框架

### 可视化：Canvas 绘制

角色关系图谱用的是原生 Canvas API 实现的**力导向图**。没有引入 D3.js 或者 vis.js 这样的重量级库，原因是我只需要一个简单的力模拟——电荷排斥 + 弹簧吸引力 + 阻尼衰减，几十行代码就能搞定。拖拽交互也直接在 Canvas 上处理了。

### 依赖一览

```
fastapi      — Web 框架
uvicorn      — ASGI 服务器
openai       — LLM 调用（兼容 DeepSeek 等）
pyyaml       — YAML 解析/序列化
python-docx  — 上传 .docx 文件时提取文本
js-yaml (CDN)— 前端 YAML 解析
```

## 架构设计（3:00 - 4:30）

**【画面：架构图——前端 → 后端 → LLM 的流程】**

整个架构很简单：**浏览器 ↔ FastAPI 后端 ↔ LLM API**，纯三层。

核心工作流程是这样的：

### 第一步：章节检测

用户贴入小说文本后，前端自动发请求到 `/api/detect-chapters`，后端用正则匹配"第X章""Chapter X"等模式，识别出文本中有多少章、每章的起始位置。

这步的作用是校验——至少需要 3 章才能转换，避免用户贴一段话就点转换浪费 API 调用。

### 第二步：流式转换

点击"转换为剧本"后，前端请求 `/api/convert-stream`，后端把整本小说发给 LLM，同时开启 **SSE 流式响应**。

```python
# StreamingResponse + Generator
async def generate():
    stream = client.chat.completions.create(
        model=LLM_MODEL,
        messages=[SYSTEM_PROMPT, user_content],
        stream=True,
    )
    for chunk in stream:
        yield f"data: {json.dumps({'type': 'chunk', 'content': delta})}\n\n"
```

前端收到 `chunk` 事件就追加显示，收到 `done` 事件就完成。用户能实时看到 LLM 逐字生成剧本的过程，体验比干等好得多。

### 第三步：逐章转换（长文本方案）

对于超过 5 章的长篇小说，还有一个**逐章转换**模式。后端按章节拆分小说，逐章调用 LLM，每章独立转换完成后，后端用 `merge_chapter_yamls()` 做**增量合并**：
- 角色去重（以 id 为准）
- 场景重新编号
- 合并 meta 信息

这样就不会因为小说太长超出 LLM 的上下文窗口。

### 第四步：输出与呈现

转换完成后，YAML 剧本会展示在右侧面板。用户可以做几件事：
- **编辑**：直接修改 YAML 源码，前端实时校验语法
- **预览**：解析 YAML 渲染成场景卡片，展示每场的环境、动作、对白
- **角色**：独立展示角色信息+人物关系
- **关系图**：力导向图可视化角色关系网
- **下载**：导出 YAML 原始文件或 Markdown 格式的剧本

## 核心功能演示（4:30 - 7:00）

**【画面：屏幕录制，实际操作】**

现在我来实际演示一下。

### 1. 输入小说文本

首先加载一段示例文本——这是一个现代都市爱情小说的第一章。可以看到左上角自动检测到了章节信息，"3 章，可转换"。

### 2. 转换为剧本

输入 API Key，点击"转换为剧本"。右侧会实时流式输出 YAML 内容。

可以看到 LLM 正在逐字生成：
- meta 块：剧本元信息
- characters：自动提取的角色——林晚（主角）、顾深（主角）、陈姐（配角）
- scenes：按地点和时间划分的场景——咖啡店、不同时间段

### 3. 预览模式

切换到"预览"标签。可以看到 YAML 被解析成了一张张场景卡片。每张卡片包含：
- 场景编号和地点
- 环境描述（斜体）
- 出场角色标签
- 动作序列
- 对白气泡——说话人、情绪、台词正文

这种展示方式比直接看 YAML 直观得多，非技术用户也能理解。

### 4. 角色视图

切换到"角色"标签。所有角色以独立卡片展示，包含性格特征、外貌描述、角色关系。

### 5. 关系图谱

切换到"关系图"标签。这是一个力导向图——角色是节点，关系是连线。绿色是正面关系，红色是负面关系，灰色是中性关系。节点可以拖拽，位置会自动调整。

这里没有用 D3.js，就是纯粹的 Canvas API + 力模拟算法：节点之间有库伦斥力，有关系的节点之间有弹簧引力，再加一个阻尼系数让系统收敛。

### 6. 导出

点击"下载 → Markdown 剧本"，可以把剧本导出为传统剧本格式，方便打印或者导入其他写作软件。

## 关键技术挑战与解决（7:00 - 8:30）

**【画面：代码片段 + 错误信息截图】**

在开发过程中遇到几个有意思的技术挑战。

### 挑战一：LLM 生成的 YAML 不合法

这是最大的坑。LLM 虽然擅长生成结构化文本，但偶尔会输出不合法的 YAML。最常见的问题是：

1. **Dialogue 内出现 `actions:` 数组**（正确做法是只用 `action: "字符串"`）
2. **同一层级出现重复的 mapping key**（LLM 可能在场景级别生成两个 `actions:`）

**解决方案**：写了一个两阶段的 YAML 修复函数。
- **Pass 1**：状态机逐行扫描，识别 dialogue 上下文，移除 dialogue 块内错误的 `actions:` 字段
- **Pass 2**：跟踪每个缩进层级的 key 名，发现重复 key 就跳过该行及其所有子行

这个函数在前后端各实现了一遍——后端 Python 版在 LLM 输出后自动修复，前端 JS 版在用户编辑时实时校验修复。

### 挑战二：前端 YAML 解析与后端不一致

前端用 js-yaml，后端用 PyYAML，两个库对错误 YAML 的容忍度不同。PyYAML 默认允许重复 key（后出现的覆盖先出现的），而 js-yaml 会直接报错。

这就导致了一种情况：后端通过了校验，但前端预览标签页切换时崩溃。

**解决方案**：让两端的修复逻辑保持一致，同时前端解析失败时自动调后端的 `/api/fix-yaml` 接口兜底。

### 挑战三：长文本的上下文窗口限制

LLM 的上下文窗口有限。对于长篇小说，一次性发送会超出限制。

**解决方案**：实现了"逐章转换"模式——按章节拆分，逐章调用 LLM，后端增量合并。每一章的转换结果独立推送给前端，用户可以实时看到每一章的完成进度。

## 设计理念（8:30 - 9:30）

**【画面：Schema 文档截图】**

在数据模型设计上，有几个有意思的决策。

### 为什么角色表独立于场景？

角色的信息（性格、外貌、关系）是整个故事级别的数据。如果散落在每个场景里，就会大量重复，而且修改起来很麻烦。所以我把角色集中定义在 `characters` 数组，场景中只通过 `id` 引用。

这样做的好处是：可以单独展示"角色关系图谱"，可以快速回答"某个角色在哪些场景中出现"这类问题。

### 为什么 emotions 用自由文本而非枚举？

情绪描述是高度依赖上下文的——"压抑着愤怒的微笑"这种表达，枚举无法覆盖。LLM 擅长生成细粒度描述，限制成枚举反而损失信息。

### 为什么保留 parenthetical（括号说明）？

括号说明是好莱坞剧本格式的标准组成部分。它和 emotion 是互补的——emotion 是宏观情绪，parenthetical 是具体的表演指引，比如"讽刺地""低声地"。
